## PptxGenJS 관련

### L-001: mkShadow는 반드시 팩토리 함수로
- **상황**: PptxGenJS가 shadow 객체를 내부에서 mutate하여, 같은 객체를 재사용하면 2번째 슬라이드부터 shadow가 깨짐
- **규칙**: `const mkShadow = () => ({ ... })` — 매번 새 객체 반환하는 팩토리 사용
- **발생**: V5 세션

### L-002: IMG 헬퍼에서 prefix 중복 주의
- **상황**: `sharp`로 생성한 base64에 이미 prefix가 붙어있는 경우 이중 prefix로 이미지 깨짐
- **규칙**: `genIcon()`은 base64만 반환, `IMG()` 헬퍼가 "image/png;base64," prefix 추가
- **발생**: V5 세션

### L-003: 팀 프로필 이미지 경로 검증
- **상황**: 파일명을 추측하여 코드에 넣었더니 실제 파일이 다른 이름이었음
- **규칙**: Glob으로 실제 파일 목록을 먼저 확인한 후 경로를 지정할 것
- **발생**: V6 세션

### L-004: 슬라이드 요소 대비(Contrast) 검증 필수
- **상황**: `midGray`(8B95A5)를 밝은 배경에 사용하여 대비 부족. QA에서 발견.
- **규칙**: 밝은 배경에는 `bodyText`(2D3748) 또는 `subText`(64748B) 사용. `midGray`는 dark 배경 전용.
- **수정 방향**: 밝은 카드 위 텍스트 → `subText`, 퍼센트·라벨 → `bodyText`
- **발생**: V8 세션

### L-005: color 이름과 실제 명도 혼동 주의
- **상황**: `lightGray`(E8EBF0)를 "밝은 배경 위의 텍스트를 더 잘 보이게" 하려고 적용 → 오히려 대비 악화
- **규칙**: 색상 변경 시 hex 값의 실제 명도를 확인. 이름에 속지 말 것.
- **발생**: V8 세션

## 인프라 / 환경 관련

### L-006: WebFetch는 egress proxy에 의해 차단될 수 있음
- **상황**: 외부 URL WebFetch 시도 → 차단. WebSearch는 작동.
- **규칙**: 웹 콘텐츠 수집 시 WebSearch 우선 사용. WebFetch 실패 시 대안으로 전환.
- **발생**: V7 세션

### L-007: LibreOffice PDF 변환 시 한국어 폰트 미지원
- **상황**: soffice로 PDF 변환 후 한글이 "????"로 표시됨
- **규칙**: 이것은 VM 환경 제한이며 PPTX 자체 문제가 아님. QA 시 한국어 렌더링 이슈는 무시. Content QA(markitdown)로 텍스트 정확성 별도 검증.
- **발생**: V8 세션

### L-008: soffice PDF 출력 경로 주의
- **상황**: `--convert-to pdf`가 입력 파일과 같은 디렉토리가 아닌 CWD에 출력하는 경우 있음
- **규칙**: 변환 후 `ls`로 실제 출력 위치를 확인한 후 후속 작업 진행
- **발생**: V8 세션

## 작업 흐름 관련

### L-009: 병렬 도구 호출 시 첫 번째 호출 실패 가능
- **상황**: 같은 메시지에서 WebSearch와 다른 도구를 동시 호출하면 WebSearch가 실패
- **규칙**: WebSearch는 단독 호출하거나, 실패 시 재시도할 것
- **발생**: V7 세션

### L-010: 컨텍스트 소진 관리
- **상황**: 긴 세션에서 컨텍스트 창이 소진되어 이전 작업 내용을 잃음
- **규칙**: 서브에이전트를 적극 활용하여 메인 컨텍스트를 가볍게 유지. 중간 결과물은 파일로 저장.
- **발생**: V7→V8 세션 전환

### L-011: QA는 생성 직후 반드시 수행
- **상황**: PPTX 생성 후 QA 없이 완료 선언 → 사용자가 직접 이슈 발견
- **규칙**: PPTX 생성 → PDF 변환 → 이미지 추출 → Visual QA(서브에이전트) + Content QA(markitdown) 파이프라인을 항상 실행
- **발생**: 초기 V5 세션

## 콘텐츠 관련

### L-012: 재무 데이터 정확성 — ARR ≠ 매출
- **상황**: CTO 피드백 — ARR(연간반복매출)과 매출(Revenue)을 혼용
- **규칙**: ARR은 구독 기반 반복 매출만 포함. 일회성 매출은 별도 표기.
- **발생**: V7 세션 (CTO 피드백)

### L-013: 안전 표현 — "100% 보장" 금지
- **상황**: CEO 피드백 — "Pevo도 정확성 100% 보장 못함, 위험하면 약사 방문 안내"
- **규칙**: AI 정확도는 "95%+ 목표" 등 목표치로 표현. "보장"이라는 표현 절대 사용 금지.
- **발생**: V7 세션 (CEO 피드백)

## IR 피치덱 관련

### L-014: 수정 전 반드시 팀 검토 → 의견 논의 → 반영 프로세스
- **상황**: MungPay IR V2 Slide 2 수정을 팀 검토 없이 바로 반영
- **규칙**: 피치덱 수정 요청 시 → 관련 팀 검토 → 찬반/보완 의견 논의 → 합의 후 반영. 바로 고치지 말 것.
- **발생**: MungPay V2 세션 (2026.03.13, CEO 피드백)

### L-015: Problem 슬라이드는 모든 이해관계자의 리스크를 포함해야
- **상황**: Slide 2에서 보호자 리스크만 다뤘는데, 가맹점주·펫시터도 리스크를 부담
- **규칙**: B2B+B2C 모델이면 Problem에서 양쪽 페인포인트를 모두 다룰 것. 한쪽만 다루면 솔루션 범위가 좁아 보임.
- **발생**: MungPay V2 세션 (2026.03.13, CEO 피드백)

### L-016: SOM 산출식은 반드시 직접 계산하여 검증
- **상황**: V3 SOM을 "~200억"으로 기재했으나, 실제 계산(500×3,500만×3.5%×12) = 73.5억. 3개 팀이 동시 발견.
- **규칙**: 산출식을 기재할 때 반드시 직접 계산하여 결과값과 일치하는지 검증할 것. 특히 투자자가 직접 계산할 수 있는 숫자는 오류 시 신뢰도 치명타.
- **발생**: MungPay V3 → V3.1 세션 (2026.03.14, 향상된 팀 리뷰)

### L-017: 보험 용어는 보험업법 기준 정확한 상품 분류명 사용
- **상황**: "단기보험"은 보험 종류가 아닌 기간 유형. 보험업법상 "소액단기보험" 또는 "배상책임보험" 등이 정확.
- **규칙**: 보험 관련 용어는 보험업법/감독규정 기준 정식 명칭 사용. 파트너 보험사와 협의된 상품명이 있으면 그것을 사용.
- **발생**: MungPay V3 → V3.1 세션 (2026.03.14, 토론팀 전문가·법무팀 발견)

### L-018: 기술 용어(AI Agent 등)는 한 곳에 정의하고 전체 참조
- **상황**: "AI Agent"가 Solution, Product, Business Model, Growth에서 각각 다른 맥락으로 사용되어 혼란.
- **규칙**: 기술 용어는 최초 등장 시 명확히 정의하고, 이후 슬라이드에서는 참조 구조로 통일.
- **발생**: MungPay V3 → V3.1 세션 (2026.03.14, 기획팀 요구사항 분석가 발견)
