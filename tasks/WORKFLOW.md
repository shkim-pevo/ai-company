# Pevo — 워크플로우 규칙

> 이 문서는 모든 세션에서 따라야 할 작업 규칙이다.
> 세션 시작 시 `lessons.md`와 함께 반드시 읽는다.

---

## 1. 기본은 계획 모드

- 사소하지 않은 모든 작업은 계획 모드로 시작 (3단계 이상 또는 구조적 결정 필요 시)
- 뭔가 잘못되면 즉시 멈추고 다시 계획 — 밀어붙이지 말 것
- 계획 모드는 개발뿐 아니라 검증 단계에서도 활용
- 모호함을 줄이려면 처음부터 상세 스펙을 작성할 것

## 2. 서브에이전트 활용

- 메인 컨텍스트 창을 깔끔하게 유지하려면 서브에이전트를 적극 활용
- 조사, 탐색, 병렬 분석은 서브에이전트에 맡길 것
- 복잡한 문제일수록 서브에이전트를 더 많이 투입
- 서브에이전트 하나당 하나의 작업만 — 집중 실행

## 3. 자기개선 루프

- 사용자에게 수정을 받을 때마다: 해당 패턴을 `tasks/lessons.md`에 기록
- 같은 실수를 반복하지 않도록 스스로 규칙을 작성
- 실수율이 낮아질 때까지 교훈들을 반복해서 다듬을 것
- **세션 시작 시 해당 프로젝트의 교훈 목록을 먼저 검토**

## 4. 완료 전 반드시 검증

- 작동한다는 걸 증명하기 전까지 절대 작업 완료로 표시하지 말 것
- 필요한 경우 main과 변경 사항 간의 동작 차이를 비교
- 스스로에게 물어볼 것: **"시니어 엔지니어가 이걸 승인할까?"**
- 테스트 실행, 로그 확인, 정확성 입증

## 5. 우아함을 추구할 것 (균형 있게)

- 사소하지 않은 변경이라면: 잠깐 멈추고 "더 우아한 방법이 있지 않을까?" 자문
- 수정이 어설프게 느껴진다면: "지금 내가 아는 모든 걸 바탕으로, 우아한 해결책을 구현해"
- 단순하고 명백한 수정엔 이 과정을 생략 — 오버엔지니어링 금지
- 결과물을 내놓기 전에 스스로 검토

## 6. 자율적인 버그 수정

- 버그 리포트를 받으면: 그냥 고칠 것. 하나하나 물어보지 말 것
- 로그, 에러, 실패한 테스트를 직접 찾아서 해결
- 사용자가 컨텍스트를 전환할 필요가 없도록
- 어떻게 하라는 말 없이도 실패한 CI 테스트를 직접 가서 고칠 것

## 작업 관리 프로세스

```
계획 → 배정 → 실무 → 팀장 검토 → 검증 → 문서화 → 교훈 기록
```

1. **계획 먼저**: 체크 가능한 항목으로 `tasks/todo.md`에 계획 작성
2. **담당자 배정**: 팀장이 적합한 담당자에게 실무 배정
3. **담당자 실무 수행**: 담당자가 독립적으로 분석/개발/작성 수행
4. **팀장 검토**: 팀장이 산출물 품질 검증, 피드백, 보완 지시
5. **진행 상황 추적**: 진행하면서 완료 항목 표시
6. **결과 문서화**: `tasks/todo.md`에 검토 섹션 추가
7. **교훈 기록**: 수정 후 `tasks/lessons.md` 업데이트

## 팀 내 역할 분담 원칙

| 역할 | 책임 | 산출물 |
|------|------|--------|
| **담당자 (실무)** | 실제 분석, 개발, 작성, 조사 수행 | 초안, 분석 결과, 코드, 데이터 |
| **팀장 (검토)** | 산출물 품질 검증, 방향성 확인, 최종 승인 | 검토 의견, 승인/반려, 보완 지시 |

```
[업무 수령] → 팀장이 담당자 배정 → 담당자 실무 수행 → 팀장 검토/검증 → 보고
                                        ↑                    │
                                        └── 보완 필요 시 ────┘
```

**핵심 규칙:**
- 담당자가 1차 산출물을 만들고, 팀장은 검토자로서 품질을 보증한다
- 팀장은 직접 실무를 하지 않되, 방향 설정과 최종 판단에 책임진다
- 복수 담당자가 있는 경우 병렬 수행 후 팀장이 통합 검토한다

## 핵심 원칙

| 원칙 | 설명 |
|------|------|
| **단순함 우선** | 모든 변경은 최대한 단순하게. 건드리는 코드는 최소화. |
| **게으름 금지** | 근본 원인을 찾을 것. 임시방편 없음. 시니어 개발자 기준. |
| **최소 영향** | 꼭 필요한 부분만 수정. 새로운 버그를 만들지 말 것. |

## 실행 규칙 (구체적 메커니즘)

### 계획 모드 스펙 작성
- 상세 스펙은 `tasks/todo.md`의 "현재 진행 중" 섹션에 체크리스트로 작성
- 스펙 수준: 각 단계별 입력 → 처리 → 출력 + 검증 기준을 명시
- 3단계 이상이면 계획 모드 진입, 그 미만이면 바로 실행

### 서브에이전트 결과 통합
- 서브에이전트 완료 후 결과 요약을 메인 컨텍스트에 1~2줄로 기록
- 병렬 작업 결과는 파일로 저장 → 메인에서 파일 참조
- 서브에이전트 실패 시: 멈추고 실패 원인 파악 → 재할당 또는 계획 재수립

### 완료 기준 (체크리스트)
- [ ] 생성물이 의도한 대로 작동하는가 (테스트/실행 확인)
- [ ] 로그에 에러 없는가
- [ ] Content QA 통과 (텍스트 정확성)
- [ ] Visual QA 통과 (레이아웃·대비·정렬)
- [ ] "시니어 엔지니어가 이걸 승인할까?" — Yes인가
- 위 항목 모두 통과 후에만 `[x]` 완료 표시

### 우아함 vs 단순함 판단 기준
- 핫픽스(1~2줄 수정): 단순함 우선 → 바로 수정
- 구조적 변경(3줄+ 또는 여러 파일): 우아함 자문 → 더 나은 방법 있으면 채택
- 판단 불가 시: 단순함이 기본값

## 커뮤니케이션 규칙

### 워크플로우 투명성
모든 작업에서 단계별로 진행 상황을 중간중간 대표님께 공유한다.

- **지금 단계** — 현재 어떤 단계에 있는지
- **하는 이유** — 왜 이 작업을 하는지
- **다음 단계** — 이 다음에 무엇을 할 건지
- **판단 필요 지점** — 대표님의 결정이 필요한 부분

형식 예시:
> 📍 **[단계 N/M] 제목** — 설명

### 영어 사고 → 한국어 결과물
작업 품질을 높이기 위해 두 단계로 진행한다.

1. **내부 분석**: 핵심 논지와 근거를 영어로 먼저 정리 (논리적 정밀도 향상)
2. **결과물 전달**: 최종 결과물과 대표님께 공유하는 내용은 한국어로 작성

## IR 피치덱 워크플로우 (MungPay V3.1 세션에서 도출)

### 수정 프로세스 (L-014)
```
수정 요청 접수 → 관련 팀 검토 → 찬반/보완 의견 논의 → 합의 후 반영
```
- 피치덱 수정을 바로 반영하지 말 것. 반드시 팀 리뷰를 거칠 것.

### 숫자 검증 필수 (L-016)
- 산출식을 기재할 때 반드시 **직접 계산하여 결과값과 일치하는지 검증**
- 특히 투자자가 직접 계산할 수 있는 숫자(TAM/SAM/SOM, 런웨이, 유닛이코노믹스)

### 전문 용어 정확성 (L-017, L-018)
- 보험/법률/규제 용어는 관련 법령 기준 정식 명칭 사용
- 기술 용어(AI Agent 등)는 최초 등장 시 명확히 정의, 이후 참조 구조로 통일

### 크로스 문서 일관성
- IR 피치덱 수정 후 반드시 관련 문서 동기화: `사업계획서.md`, `skill-dashboard.md`, `lessons.md`
- 숫자(TAM/SOM/매출/런웨이)가 여러 문서에 걸쳐 있으면 **모든 출처를 동시에 업데이트**
